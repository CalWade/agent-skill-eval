# agent-skill-eval 项目产出说明

## 一、情景来源

### 问题背景

在 AI Agent 平台（如 OpenClaw）中，Agent 通过 SKILL.md 文档学习如何调用外部 API 完成用户任务。但在实际使用中，我们遇到了一个核心矛盾：

> **SKILL.md 写得"看起来对"，但 Agent 跑起来经常失败，而且失败原因五花八门——无法系统性定位和改进。**

具体表现：

1. **52 步才找到一个人。** 在实际测试"给黄威健发一条消息"时，Agent 执行了 52 次工具调用。原因是 SKILL.md 没有明确优先使用"邮箱反查 API"，Agent 退化为逐个遍历部门树。理论最少 3 步，实际步骤效率仅 5.8%。
2. **失败原因分散且不可追溯。** 权限没开通、Agent 自己编流程、API 返回格式变了、用户指令太模糊——这些问题混在一起，无法区分是 SKILL.md 的问题还是 Agent 模型的问题。
3. **优化缺乏数据支撑。** 改了 SKILL.md 之后效果是好了还是坏了？没有量化手段衡量。改一个 skill 可能影响另一个 skill，全靠人工回归。

### 直接触发点

在网页端手动测试飞书发消息功能时，发现 Agent 的通讯录查找行为极度低效（52 步遍历部门树）。这暴露出一个系统性问题：**没有自动化的评估工具来持续检测 SKILL.md 的质量。**

---

## 二、设计思路

### 核心设计原则

**"用测试用例驱动 SKILL.md 优化，用数据说话。"**

整个框架的设计围绕三个关键决策：

### 1. 平台无关——兼容任何 OpenAI Chat Completions 格式的 Agent API

不绑定 OpenClaw、Dify、FastGPT 等任何特定平台。只要 Agent 平台暴露了标准的 Chat Completions API，就能直接接入。通过 `.env` 配置 `AGENT_API_URL` + `AGENT_API_KEY` + 可选的 `AGENT_EXTRA_BODY`（JSON 格式传入平台特有参数），实现零代码适配。

### 2. YAML 驱动——测试用例即配置

测试用例不写代码，全部用 YAML 声明式定义。降低编写门槛的同时，保证了可读性和可维护性。一份 YAML 同时服务于：

- **自动化执行**：`run_eval.py` 直接解析运行
- **人工 review**：结构清晰，非工程师也能理解和修改
- **文档生成**：可以从 YAML 自动生成 Markdown 格式的用例说明

### 3. 三层递进判定——解决 LLM 回复措辞不可预测的问题

这是本框架的核心技术点。LLM 回复的措辞高度不确定——"已发送"、"发送成功"、"消息已送达"、"✅ Done"表达的是同一个语义。单一的精确匹配必然导致大量误判。

解决方案是 `semantic_success` 三层递进：

```
关键词匹配（零成本）→ 正则匹配（低成本）→ LLM-as-Judge（高精度）
```

- **第一层：关键词匹配。** 零 API 调用成本，覆盖 80% 的常见表述。
- **第二层：正则匹配。** 覆盖措辞变体（如"发送.*成功"、"消息.*完成"），仍然零成本。
- **第三层：LLM-as-Judge。** 只在前两层都未命中时触发，调用另一个 LLM 做语义判定。兼顾了准确性和成本。

三层中任一层通过即判定为 PASS，避免了因措辞差异导致的误判，同时把 LLM 裁判调用量控制在最低限度。

### 评估维度体系（D1-D5）

不只看"能不能跑通"，而是从五个维度量化 Agent 技能的质量：

| 维度 | 衡量什么 | 为什么重要 |
|------|---------|-----------|
| D1 首次成功率 | 一次跑通，不需人工干预 | 用户体验的第一指标 |
| D2 步骤效率 | 实际步骤 vs 理论最少步骤 | 冗余步骤 = 浪费 token = 更容易卡死 |
| D3 错误恢复率 | 遇到 API 错误后能否自动修复 | 零基础用户的生命线 |
| D4 输出质量 | 回复是否完整、准确、可操作 | 任务完成 ≠ 用户满意 |
| D5 上下文消耗 | Token 用量和响应轮次 | 直接影响成本和是否会卡死 |

### 失败分类体系（F1-F7）

每次测试失败时标记根因分类，数据积累后可以精准定位最高频的失败类型，指导优化方向：

| 代码 | 分类 | 示例 |
|------|------|------|
| F1 | Skill 文档缺陷 | SKILL.md 流程有遗漏 |
| F2 | Agent 理解偏差 | Agent 没按文档执行 |
| F3 | 权限/配置问题 | 权限未开通 |
| F4 | API 限制 | 接口本身有 bug |
| F5 | 上下文溢出 | Token 超限 |
| F6 | 用户指令歧义 | 指令太模糊 |
| F7 | 跨 Skill 依赖失败 | 联动其他 Skill 时失败 |

## 三、实现目标与交付物

### 已完成的交付物

#### 1. 评估引擎（`src/`）

- **`run_eval.py`**（~350 行）：核心测试执行器
  - 解析 YAML 测试用例
  - 调用任意 OpenAI 兼容 Agent API（stream=false）
  - 七种判定类型（`semantic_success`、`output_contains`、`llm_judge`、`duration_le` 等）
  - 自动重试机制（429 限频、5xx 服务端错误）
  - 双格式报告输出（Markdown + JSON）
  - 支持 `--dry-run` 预览模式

- **`gen_summary.py`**（~120 行）：跨轮次汇总看板
  - 自动扫描 `results/` 下所有 JSON 结果
  - 生成按 suite 聚合的成功率表格
  - 跨轮次趋势对比（↑/↓ 箭头）
  - 近期失败记录汇总

#### 2. 方法论文档（`docs/`）

- **`eval-dimensions.md`**：D1-D5 五维度定义 + F1-F7 失败分类法
- **`improvement-log.md`**：数据驱动的优化记录模板（含第一条实际发现：52 步遍历问题）

#### 3. 实战示例（`examples/openclaw-feishu/`）

以 OpenClaw 飞书技能为实际案例：
- 3 个代表性 SKILL.md 定义（发消息、创建文档并授权、电子表格）
- 3 套 YAML 测试用例（20 条用例，覆盖 happy_path / error_recovery / edge_case）
- 1 套安全冒烟测试（不触发外部操作，验证 pipeline 是否跑通）

#### 4. 工程化配置

- `Makefile`：`make setup` / `make smoke` / `make summary` 快捷命令
- `.env.example`：带详细注释的配置模板
- `.gitignore`：排除测试结果、IDE 文件、敏感配置

### 量化成果

| 指标 | 数值 |
|------|------|
| 评估引擎代码量 | ~470 行 Python |
| 支持的判定类型 | 7 种 |
| 示例测试用例 | 20 条（3 个 YAML 套件） |
| 示例 Skill 定义 | 3 个（简单/中等/复杂各一） |
| 配置项 | 11 个（2 必填 + 9 可选） |

## 四、未来展望

### 短期改进（可直接动手）

1. **Stream 模式支持**
   当前使用 `stream=false` 等待完整响应。接入 `stream=true` 后可以实时捕获 Agent 的每一步工具调用，实现真正的步骤计数（D2）和工具调用验证（`tool_used`、`step_count_le`），而非目前的"从回复文本推断"。

2. **CI/CD 集成**
   将测试作为 GitHub Actions 的质量卡点。每次 SKILL.md 变更自动触发回归测试，成功率低于阈值则阻断合并。

3. **更多 Skill 的测试用例**
   当前只覆盖了 3 个飞书 skill 的用例。扩展到日历、审批、多维表格等场景，提升回归测试的覆盖面。

### 中期扩展（需要更多投入）

4. **A/B 对比模式**
   支持在同一组测试用例上对比两个版本的 SKILL.md（或两个 LLM 模型），自动生成差异报告：哪些用例从 FAIL 变 PASS，哪些退化了。

5. **交互式多轮测试**
   当前只支持单轮指令。扩展为多轮对话测试（如先查人再发消息），验证 Agent 的上下文保持能力和跨 Skill 联动能力。

6. **可视化 Dashboard**
   将 JSON 结果接入 Web 看板（如 Streamlit 或 Grafana），实时展示成功率趋势、失败热点分布、耗时分布等。

### 长期愿景

7. **通用 Agent 评估基准**
   将框架从"飞书技能评估"泛化为通用的 Agent 能力评估基准。覆盖工具调用准确性、多步规划能力、错误恢复能力、幻觉检测等维度，成为 Agent 开发者的标准质量工具。
