# agent-skill-eval 迭代设计方案

## 当前状态（v0.1）

一个能跑通的结果断言器：

- 发指令 → 拿最终回复 → 按规则判 PASS/FAIL → 出报告
- 只支持单轮、黑盒、stream=false
- D1（首次成功率）和 D4（输出质量）可自动化
- D2（步骤效率）、D3（错误恢复）、D5（上下文消耗）停留在方法论定义层面
- 关键词判定存在 False Positive 风险
- 无法高频回归（测试会产生真实副作用）

下面的迭代按三个版本规划，每个版本解决一组核心问题，交付可验证的代码。

---

## v0.2 — 堵漏洞：判定可信度 + 副作用隔离

> 目标：让现有的结果断言器从"能跑"变成"可信"。

### 0.2.1 否定上下文感知

**解决的问题：** 关键词命中但语义实际为否定时的 False Positive。

**改动范围：** `src/run_eval.py` 的 `evaluate_criteria` 函数。

**设计：**

在关键词匹配命中后，回查命中位置前方 30 个字符，检测是否存在否定词。存在则该关键词不算命中。

```python
NEGATION_PREFIXES = ["失败", "未能", "无法", "不能", "没有", "抱歉", "出错",
                     "错误", "异常", "拒绝", "fail", "error", "unable", "denied"]

def keyword_in_positive_context(output: str, keyword: str) -> bool:
    """关键词命中且不在否定上下文中才算有效命中"""
    idx = output.find(keyword)
    while idx != -1:
        prefix = output[max(0, idx - 30):idx]
        if not any(neg in prefix for neg in NEGATION_PREFIXES):
            return True  # 正面上下文命中
        idx = output.find(keyword, idx + 1)
    return False  # 所有命中都在否定上下文中
```

关键词层和正则层都加上这个检查。任何一次命中处于否定上下文时，降级到下一层判定。

**验证方式：** 构造 3 个对抗用例写进 `tests/test_negation.py`：
1. "发消息失败了，但文档创建成功" — 对 send-message 的判定应为 FAIL
2. "消息已成功送达" — 应为 PASS
3. "无法发送消息，权限不足" — 应为 FAIL

### 0.2.2 LLM-as-Judge Prompt 升级

**解决的问题：** 裁判只输出 PASS/FAIL，没有推理过程，无法审计也无法发现裁判自身的偏见。

**改动范围：** `src/run_eval.py` 的 `llm_judge` 和 `_call_judge_llm` 函数。

**新 Prompt 模板：**

```
你是一个严格的测试裁判。根据条件判定 AI 助手的回复是否通过。

## 判定条件
{criteria}

## AI 助手的回复
{output}

## 判定规则
1. 先分析回复内容（2-3句话）
2. 如果回复中同时出现正面词和否定词（如"失败...但...成功"），
   以与判定条件直接相关的那部分为准
3. 最后一行严格输出: VERDICT: PASS 或 VERDICT: FAIL

## 分析
```

解析时提取 `VERDICT:` 后的结论，同时把分析部分保存到判定详情中，写进报告。

**新增返回结构：**

```python
def llm_judge(output, criteria_desc) -> tuple:
    # 返回 (passed: bool|None, detail: str, reasoning: str)
    # reasoning 写入报告的 judgments，人工复盘时可看裁判推理链
```

### 0.2.3 失败原因半自动标记

**解决的问题：** F1-F7 全靠人工贴标签。

**可自动化的部分：**

在 `judge_case` 函数中，当 verdict 为 FAIL 时，扫描 Agent 回复和错误信息，按规则自动标记"疑似 F 类"：

```python
AUTO_FAIL_TAGS = {
    "F3": ["99991672", "403", "permission", "权限不足", "未授权", "Forbidden"],
    "F5": ["context_length", "token limit", "超时", "timeout", "too long"],
    "F4": ["飞书接口", "API error", "server error", "502", "503"],
}
```

匹配到特征码时，在结果 JSON 中加一个 `auto_fail_tags: ["F3"]` 字段。不匹配时留空，人工后续补充。不替代人工判断，只是提供初筛建议。

### 0.2.4 safe-smoke 与 live-test 分级

**解决的问题：** 所有用例混在一起，没有明确区分哪些可以高频跑、哪些会产生副作用。

**改动：** 在 YAML 中新增 `side_effect` 字段：

```yaml
cases:
  - id: TC-SM-01
    side_effect: none       # 无副作用，可高频跑
    # ...
  - id: TC-SM-05
    side_effect: write      # 会写入外部系统（发消息/创建文档）
    # ...
```

`run_eval.py` 新增 `--safe-only` 参数，只执行 `side_effect: none` 的用例。CI/CD 中只跑 `--safe-only`。

## v0.3 — 打开黑盒：Trace 解析 + Stream 模式

> 目标：从"只看最终回复"升级为"看到 Agent 的每一步操作"，让 D2/D5 从纸面定义变成可自动计算的真实指标。

### 0.3.1 标准化 Trace 数据结构

新建 `src/trace.py`：

```python
from dataclasses import dataclass, field
from typing import List, Optional
from abc import ABC, abstractmethod

@dataclass
class ToolCall:
    step_index: int
    tool_name: str
    arguments: dict
    result: Optional[str] = None
    duration_ms: int = 0
    error_code: Optional[int] = None   # 飞书错误码等

@dataclass
class AgentTrace:
    """一次 Agent 执行的完整轨迹"""
    tool_calls: List[ToolCall] = field(default_factory=list)
    total_steps: int = 0
    total_tokens: int = 0
    thinking_content: str = ""          # Agent 的推理过程（如果平台暴露）

    @property
    def unique_tools(self) -> set:
        return {tc.tool_name for tc in self.tool_calls}

    @property
    def error_steps(self) -> List[ToolCall]:
        return [tc for tc in self.tool_calls if tc.error_code]

    @property
    def has_recovery(self) -> bool:
        """是否发生了错误恢复：有错误步骤，且后续仍有成功步骤"""
        if not self.error_steps:
            return False
        last_error_idx = max(tc.step_index for tc in self.error_steps)
        return any(tc.step_index > last_error_idx and tc.error_code is None
                   for tc in self.tool_calls)
```

### 0.3.2 平台 Adapter 接口

```python
class TraceAdapter(ABC):
    @abstractmethod
    def parse_stream(self, response_stream) -> tuple:
        """解析 stream 响应，返回 (final_content: str, trace: AgentTrace)"""
        ...

    @abstractmethod
    def parse_response(self, response_json: dict) -> tuple:
        """解析非 stream 响应，尽可能提取 trace 信息"""
        ...
```

**首批实现两个 Adapter：**

1. **`OpenAIAdapter`** — 解析标准 OpenAI 格式。非 stream 模式下从 `message.tool_calls` 提取；stream 模式下从 SSE 事件流中逐步拼装。

2. **`OpenClawAdapter`** — OpenClaw 的 stream 格式。从实际抓包/文档中逆向其 SSE 事件格式，提取每步的工具名、参数、返回值。

Adapter 选择通过 `.env` 配置：

```bash
# .env
TRACE_ADAPTER=openclaw   # 可选: openai, openclaw, dify, none
```

`none` 表示不解析 Trace，退化为 v0.1 的纯结果断言模式，保持向后兼容。

### 0.3.3 D2/D5 自动计算

有了 `AgentTrace` 后，`evaluate_criteria` 中之前被跳过的判定类型可以真正实现：

```python
# step_count_le: 不再跳过
elif ctype == "step_count_le":
    max_steps = c["value"]
    actual = trace.total_steps
    passed = actual <= max_steps
    detail = f"步骤数 {actual} ≤ {max_steps} → {'通过' if passed else f'超出 {actual - max_steps} 步'}"

# tool_used: 真正检查工具调用
elif ctype == "tool_used":
    tool_name = c["name"]
    passed = tool_name in trace.unique_tools

```

### 0.3.4 `call_agent` 重构为双模式

当前的 `call_agent` 只支持 `stream=false`。重构为：

```python
def call_agent(instruction: str, adapter: TraceAdapter = None, timeout=300) -> dict:
    if adapter:
        return _call_agent_stream(instruction, adapter, timeout)
    else:
        return _call_agent_simple(instruction, timeout)  # 原有逻辑不动
```

stream 模式下，逐步读取 SSE 事件，交给 adapter 解析，最终返回的 result dict 中多出一个 `trace: AgentTrace` 字段。

## v0.4 — 多轮编排 + 裁判校准

> 目标：支持跨步骤的状态依赖测试，同时建立裁判 LLM 自身的质量评估机制。

### 0.4.1 多轮测试引擎

**YAML 格式扩展：**

```yaml
skill: feishu-send-message
mode: multi_turn

cases:
  - id: TC-MT-01
    title: 查人 → 发消息联动
    steps:
      - id: step-1
        instruction: "帮我查一下刘伟的 open_id"
        extract:
          user_id: 'ou_[a-zA-Z0-9]+'
        pass_criteria:
          - type: output_matches_regex
            pattern: 'ou_[a-zA-Z0-9]+'

      - id: step-2
        instruction: "给刘伟发一条消息：项目进度如何"
        depends_on: step-1             # 显式声明依赖
        pass_criteria:
          - type: semantic_success
            description: "消息发送成功"
            keywords: ["发送成功", "已送达", "消息已发送"]
```

**引擎侧改动（`src/run_eval.py`）：**

```python
def run_multi_turn(case: dict, adapter=None) -> dict:
    context = {}           # 跨步骤变量池
    step_results = []

    for step in case["steps"]:
        # 模板变量替换
        instruction = step["instruction"]
        for var_name, var_value in context.items():
            instruction = instruction.replace(f"{{{{{var_name}}}}}", var_value)

        # 执行
        result = call_agent(instruction, adapter)

        # 提取变量
        if "extract" in step and result["success"]:
            for var_name, pattern in step["extract"].items():
                match = re.search(pattern, result["output"])
                if match:
                    context[var_name] = match.group(0)

        # 判定
        judgment = judge_case(result, step)
        step_results.append({**judgment, "step_id": step["id"]})

        # 某步失败则跳过后续
        if judgment["verdict"] == "FAIL":
            for remaining in case["steps"][case["steps"].index(step)+1:]:
                step_results.append({
                    "step_id": remaining["id"],
                    "verdict": "SKIP",
                    "reason": f"前置步骤 {step['id']} 失败",
                })
            break

    all_passed = all(s["verdict"] == "PASS" for s in step_results)
    return {
        "verdict": "PASS" if all_passed else "FAIL",
        "steps": step_results,
        "context": context,
    }
```

兼容性：`mode` 字段缺省时默认为 `single_turn`，走原有逻辑，不影响已有用例。

### 0.4.3 D3 错误恢复率的自动计算

有了 v0.3 的 `AgentTrace`，D3 可以自动化：

```python
def calc_d3(trace: AgentTrace) -> dict:
    error_steps = trace.error_steps
    if not error_steps:
        return {"d3_score": None, "note": "未触发错误，D3 不适用"}

    recovery_count = 0
    for err in error_steps:
        # 检查该错误之后是否有成功的相同工具调用
        later_success = [
            tc for tc in trace.tool_calls
            if tc.step_index > err.step_index
            and tc.tool_name == err.tool_name
            and tc.error_code is None
        ]
        if later_success:
            recovery_count += 1

    d3 = recovery_count / len(error_steps) * 100
    return {
        "d3_score": d3,
        "errors_total": len(error_steps),
        "recovered": recovery_count,
    }
```

联动 D5 的综合评分：

```python
def calc_combined_score(d3_score, actual_steps, min_steps):
    d5_score = min(1.0, min_steps / max(actual_steps, 1)) * 100
    if d3_score == 0:
        return 0
    return d3_score * (0.6 + 0.4 * d5_score / 100)
```

## 版本路线图总览

```
v0.1（当前）          v0.2                v0.3                v0.4
结果断言器    →    可信断言器     →    Trace 诊断器    →    编排评测平台
                   │                   │                   │
                   ├ 否定上下文感知     ├ Trace 数据结构     ├ 多轮 steps 编排
                   ├ Judge CoT         ├ Adapter 模式       ├ 裁判校准数据集
                   ├ 半自动 Fail Tag   ├ Stream 解析        ├ D3 自动计算
                   └ side_effect 分级  ├ D2/D5 真实计算     └ D3×D5 联动评分
                                       └ 报告 Trace 摘要
```

每个版本的核心原则：

- **v0.2 不加新概念，只修已有的漏洞。** 改动集中在 `run_eval.py` 一个文件内，不引入新模块。
- **v0.3 引入一个新概念（Trace），配套新模块。** 但保持向后兼容——不配 Adapter 就退化为 v0.2 行为。
- **v0.4 在 Trace 基础上做上层能力。** 多轮编排和 D3 计算都依赖 v0.3 的 Trace 数据。

不做的事情（明确排除）：

- **不做 Web Dashboard。** 当前阶段 Markdown + JSON 报告够用，Streamlit/Grafana 是锦上添花不是雪中送炭。
- **不做录制-回放 Mock。** 需要平台侧深度配合，短期内用 `side_effect` 分级 + `--safe-only` 替代。
- **不做多裁判投票。** 成本翻倍但收益不确定，等校准数据集跑出来确认单裁判不够用时再加。
