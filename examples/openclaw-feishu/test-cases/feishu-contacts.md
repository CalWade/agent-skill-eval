# feishu-contacts 测试用例

## 依赖
- feishu-permission-setup（当权限不足时）
- feishu-manage-chat（通过群成员列表获取 open_id 的备选路径）

## 理论最少步骤
- 从缓存命中：0 步（直接返回）
- 从 mentions 提取：0 步（消息自带）
- 通过群成员列表：2 步（获取 token → 获取群成员）
- 通过通讯录搜索：2-N 步（获取 token → 遍历部门树，N 取决于部门深度）

---

## 用例列表

### TC-CT-01: 缓存命中
- **指令：** "韦贺文的 open_id 是什么"
- **前置条件：** 之前已缓存韦贺文的 open_id
- **预期结果：** 直接返回缓存值
- **Pass 标准：** 返回正确 open_id，0 次 API 调用
- **重点观察：** Agent 是否优先查缓存

### TC-CT-02: 通过通讯录按姓名查找
- **指令：** "帮我查一下刘伟的 open_id"
- **前置条件：** 刘伟不在缓存中，Bot 可用范围覆盖全员
- **预期结果：** Agent 遍历部门树找到此人
- **Pass 标准：** 返回正确的 open_id
- **重点观察：** 遍历了几层部门、用了几次 API 调用

### TC-CT-03: 重名处理
- **指令：** "查一下王伟的信息"
- **前置条件：** 通讯录中可能有多个王伟
- **预期结果：** Agent 列出所有匹配结果，让用户选择（或附带部门信息区分）
- **Pass 标准：** 不随意选一个，而是让用户确认
- **重点观察：** Agent 对歧义的处理方式

### TC-CT-04: 查找不存在的人
- **指令：** "查一下马斯克的 open_id"
- **前置条件：** 通讯录中无此人
- **预期结果：** Agent 告知未找到
- **Pass 标准：** 明确告知未找到，不编造 open_id
- **重点观察：** Agent 是否会幻觉出一个假 ID

### TC-CT-05: 通过群成员列表查找
- **指令：** "这个群里有谁，列出来"
- **前置条件：** Bot 在一个群中
- **预期结果：** Agent 调用群成员 API，返回 open_id + 姓名
- **Pass 标准：** 列表完整、open_id 正确
- **重点观察：** 是否自动缓存结果供后续使用

### TC-CT-06: 查找后自动缓存
- **指令：** 先执行 TC-CT-02，然后再次问"刘伟的 open_id"
- **前置条件：** 第一次查找成功
- **预期结果：** 第二次直接从缓存返回
- **Pass 标准：** 第二次 0 次 API 调用
- **重点观察：** 缓存机制是否生效

### TC-CT-07: Bot 可见范围受限
- **指令：** "查一下市场部张丽的信息"
- **前置条件：** Bot 可用范围未设为全员
- **预期结果：** 返回空结果或 Agent 提示需要扩大可用范围
- **Pass 标准：** 不返回错误结果，给出明确指引
- **重点观察：** Agent 是否识别到"返回用户数极少"的异常
