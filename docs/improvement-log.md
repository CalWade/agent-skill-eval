### [2026-03-10] 发现 #0：通讯录查找 52 步遍历部门树（从 API 测试数据发现）

**触发场景：** 网页端测试 "给黄威健发一条消息"
**失败分类：** F1（Skill 文档缺陷）+ F2（Agent 理解偏差）
**问题描述：**
Agent 执行了 52 次工具调用才找到黄威健的 open_id。流程如下：
1. 读 feishu-contacts SKILL.md（1步）
2. 尝试找搜索脚本但不存在（3步）
3. 找飞书配置文件（8步：.env → config.yaml → openclaw.json）
4. 获取 token（1步）
5. 从根部门开始逐个遍历子部门的成员列表（35步）
6. 最终改用邮箱反查 batch_get_id（1步）成功
7. 发送消息（1步）

**根因分析：**
1. feishu-contacts SKILL.md 的决策树把"按部门遍历"放在了可选路径里，但没有明确告诉 Agent **优先用邮箱反查**
2. SKILL.md 没有提供配置文件路径的明确指引，Agent 花了 8 步找 openclaw.json
3. 没有缓存机制的强制指引——Agent 找到 open_id 后没有保存

**改进方向：**
- [ ] feishu-contacts SKILL.md：将"邮箱/手机号反查"提升为第一优先级
- [ ] feishu-contacts SKILL.md：明确写出 openclaw.json 路径
- [ ] feishu-send-message SKILL.md：加入"找到 open_id 后自动缓存"的强制步骤
- [ ] 所有 SKILL.md：统一配置文件发现路径（避免每次 8 步摸索）

**量化指标：**
- 当前：52 步完成（理论最少 3 步）→ 步骤效率 5.8%
- 目标：优化后应在 5 步以内完成 → 步骤效率 60%+
