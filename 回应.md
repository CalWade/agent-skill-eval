# 对面试官反馈的回应

## 前置态度

先认账。"挤水分"部分说的基本都对。

产出文档确实存在包装过度的问题——把"手动发现的问题"和"自动化框架的能力"混在一起说了，给人一种框架已经能自动做步骤分析的错觉。实际上当前版本的核心能力就是"给 Agent 发指令 → 拿最终回复 → 断言回复内容"，本质是一个**结果断言器**，不是诊断中枢。这点没什么好辩的。

下面逐条回应"挤水分"部分，然后正面回答 Q1-Q4。

---

## 对"挤水分"的逐条回应

### 1. "52 步"到底怎么来的？

坦白说：**是在网页端手动数出来的**，不是框架跑出来的。

OpenClaw 的网页对话界面会实时展示每一步 Tool Call 的过程（函数名 + 参数 + 返回值），我是在页面上一步一步数的 52 次。这个数据本身是真实的，但把它写在"框架的直接触发点"里，确实容易让人误以为是框架自动检测出来的。

**修正表述：** 52 步是手动观测发现的真实问题，它驱动了框架的诞生，但框架目前不具备自动计算步骤数的能力。D2（步骤效率）在当前版本中是**方法论层面的定义**，不是代码层面的实现。产出文档应该在 D2 旁边标注"当前版本：仅定义，未自动化"。

### 2. 三层递进判定的 False Positive 问题

面试官举的例子非常精准：

> "很抱歉，给黄威健发消息**失败**了，系统提示没有权限，但测试文档创建**成功**。"

如果关键词列表里有"成功"，第一层直接 PASS——这确实是个严重的 False Positive。

**承认问题，同时说明设计时的取舍：**

当前的三层递进是为了解决一个更高频的实际问题——**False Negative（假阴性）**。在早期手动测试中，Agent 明明完成了任务，但回复"消息已送达"而非"发送成功"，如果用精确匹配就会判 FAIL。三层递进优先解决的是"别把对的判成错的"，而不是"把错的判成对的"。

但这不意味着 False Positive 可以不管。面试官说得对，在 Agent 编排场景下，一个假阳性比十个假阴性更危险——它会让你以为系统正常，实际上早就坏了。

**具体的改进方案：**

- **关键词层加入否定词前缀过滤。** 在匹配到关键词之前，检查前 N 个字符是否包含否定词（"失败"、"未能"、"无法"、"抱歉"、"错误"）。如果关键词出现在否定上下文中，不算命中，直接降级到下一层。这个改动大概 10 行代码。
- **正则层要求配合上下文。** 不是 `.*成功`，而是要求 `(发送|消息).*(成功|完成)` 这种带主语约束的正则。用例编写规范里应该强制要求正则包含操作动词。
- **用例编写侧也要改。** `semantic_success` 的 `keywords` 不应该放单独的"成功"，而应该放"发送成功"、"消息已送达"这种带上下文的短语。关键词太短是用例写得不好，不全是框架的锅。

### 3. F1-F7 是人工标签，不是自动诊断

**完全认。** 当前代码里没有任何自动分类 F1-F7 的逻辑。

F1-F7 的设计意图是给**人工复盘提供标准化的分类框架**，而不是让代码自动判断。当一个用例 FAIL 时，工程师在 `improvement-log.md` 里手动标注属于 F 几，积累到一定数量后统计哪类失败最多，优先修哪类。

产出文档里确实没有明确区分"方法论层"和"代码实现层"，容易让人以为这些都是代码自动跑出来的。这是写文档的问题。

**能做到什么程度的自动化：**

- F3（权限/配置问题）可以半自动——Agent 回复中出现 `99991672`、`403`、`permission denied` 等特征码时，自动标记为"疑似 F3"。
- F5（上下文溢出）可以自动——请求超时或返回 `context_length_exceeded` 时直接标记。
- F1/F2/F6/F7 在纯黑盒模式下几乎无法自动区分，必须看 Agent 的中间推理过程（思维链）才能判断。

### 4. 测试环境的副作用污染

这个问题在实际测试中已经遇到了——跑发消息测试时同事确实收到了真实消息。

**当前的应对（不完美但可用）：**

- `safe-smoke.yaml` 就是为此设计的：5 条用例全部是"打招呼"、"问能力"、"知识问答"，**不触发任何外部 API 操作**。用于验证 pipeline 本身是否能跑通。
- 真正需要调飞书 API 的用例（`feishu-send-message.yaml`）定位为**低频手动触发**，不放 CI/CD 里高频跑。

**但面试官说的对，如果要做 CI/CD 级别的高频回归，必须解决 Mock 问题。** 这确实是文档里完全没提的盲区。两个方向：

- **Dry-run 模式的深化：** 不是只预览用例，而是让 Agent 真正执行推理但把所有 Tool Call 拦截掉，只验证"Agent 是否选择了正确的工具和参数"而不是"工具是否真的执行成功了"。这需要平台侧配合提供 sandbox 模式。
- **录制-回放（Record-Replay）：** 第一次真实跑通后，把 Agent 的完整 Tool Call 链和 API 返回值录制下来。后续回归测试时，用录制的返回值替代真实 API 调用。只要 Agent 的推理路径没变，就不需要真正请求飞书。

## 正面回答 Q1-Q4

### Q1：跨平台的 Trace 解析协议怎么设计？

现状是：不同平台暴露 Tool Call 的方式完全不同。

- OpenAI 标准格式：`choices[0].message.tool_calls[]`，每个 tool_call 有 `function.name` + `function.arguments`
- OpenClaw：stream 模式下逐步推送，每步包含工具名、参数、返回值
- Dify：通过 workflow 日志暴露，格式是自定义的 JSON
- FastGPT：有独立的运行日志 API

**设计思路：Adapter 模式 + 标准化中间表示。**

定义一个统一的 Trace 数据结构：

```python
@dataclass
class ToolCallTrace:
    step_index: int          # 第几步
    tool_name: str           # 工具/函数名
    arguments: dict          # 调用参数
    result: str              # 返回值
    duration_ms: int         # 该步耗时
    timestamp: float         # 时间戳
```

然后为每个平台写一个 Adapter，负责把平台特有的响应格式转换为 `List[ToolCallTrace]`：

```python
class TraceAdapter(ABC):
    @abstractmethod
    def parse(self, raw_response) -> List[ToolCallTrace]: ...

class OpenAIAdapter(TraceAdapter): ...
class OpenClawStreamAdapter(TraceAdapter): ...
class DifyWorkflowAdapter(TraceAdapter): ...
```

框架核心逻辑只依赖 `List[ToolCallTrace]`，不关心底层平台。新接入一个平台只需要写一个 Adapter，不动核心代码。

**现实的妥协：** 有些平台（比如某些 Dify 部署）压根不暴露 Tool Call 过程，只给最终结果。这种情况下 Adapter 返回空列表，D2/D5 自动标记为"不可评估"而不是硬算一个假数据。

### Q2：LLM-as-Judge 的幻觉和偏见怎么破？

三个层面：

**1. Prompt 设计——强制结构化输出 + CoT**

当前的裁判 Prompt 太简单了（"只回答 PASS 或 FAIL"），没有要求输出推理过程。改进版：

```
你是一个严格的测试裁判。

## 判定条件
{criteria}

## Agent 回复
{output}

## 要求
1. 先用 2-3 句话分析回复内容是否满足条件
2. 特别注意：如果回复中包含"失败"、"错误"、"无法"等否定表述，
   即使同时包含"成功"等正面词汇，也应判定为 FAIL
3. 最后一行只输出：VERDICT: PASS 或 VERDICT: FAIL

## 你的分析
```

用 CoT 强制裁判先给理由再给结论，一方面提高判定准确率，另一方面理由可以写入测试报告，人工复盘时知道裁判是怎么想的。

**2. 裁判校准——用标注数据集评估裁判本身**

收集一批人工标注的 `(Agent回复, 判定条件, 人工PASS/FAIL)` 三元组，定期跑一遍裁判，算裁判和人工标注的一致率。如果一致率低于 90%，说明裁判 Prompt 需要调整或者换模型。

这本质上是用传统 ML 评测的方式评测裁判 LLM——把裁判当成一个二分类器，算 Precision/Recall/F1。

**3. 多裁判投票（成本换准确率）**

对高优先级用例（P0 级别），用两个不同的 LLM 分别判定，只有两者一致才给最终结论，不一致时标记为"需人工复核"。成本翻倍但置信度显著提升。日常跑不需要，发版前的关键回归可以开。

### Q3：YAML 怎么处理状态依赖？

面试官说的对，纯 YAML 处理不了动态变量传递。我的方案是**不硬塞进 YAML**，而是在 YAML 之上加一层轻量编排。

**方案：YAML 定义步骤，引擎管理上下文。**

```yaml
skill: feishu-send-message
mode: multi_turn  # 标记为多轮测试

cases:
  - id: TC-MT-01
    title: 查人 + 发消息的联动测试
    steps:
      - instruction: "帮我查一下刘伟的 open_id"
        extract:
          user_id: 'ou_[a-zA-Z0-9]+'   # 用正则从回复中提取，存入变量
        pass_criteria:
          - type: output_matches_regex
            pattern: 'ou_[a-zA-Z0-9]+'

      - instruction: "用 {{user_id}} 给刘伟发一条消息：项目进度如何"
        pass_criteria:
          - type: semantic_success
            description: "消息发送成功"
```

关键设计：
- `extract` 字段用正则从当前步骤的回复中提取变量，存入引擎维护的 context dict
- 后续步骤的 `instruction` 中用 `{{variable}}` 引用（就是 Mustache 模板语法，不需要引入 Jinja2 那么重的东西）
- 引擎按 `steps` 顺序执行，每步执行完更新 context，某步 FAIL 则整个 case 标记为 FAIL 并跳过后续步骤

这比引入 Lua/Python AST 轻量得多，YAML 的声明式特性没被破坏，非工程师仍然能读懂用例。变量提取用正则足够——Agent 回复中的 open_id、document_id、message_id 都有明确的格式特征。

### Q4：D3（错误恢复率）和 D5（上下文消耗）怎么联动？

面试官的核心问题是：Agent 自动恢复了但烧了 50 步 token，这到底算好还是算差？

**不应该用单一分数，应该用矩阵。**

```
                D5 消耗低        D5 消耗高
D3 恢复成功    ✅ 最优            ⚠️ 笨但能用
D3 恢复失败    ❌ 快速失败        💀 最差
```

四个象限各有含义：

- **恢复成功 + 消耗低：** 最理想。Agent 识别了错误，用最少步骤修复。说明 SKILL.md 的错误处理流程写得好。
- **恢复成功 + 消耗高：** 能用但低效。典型场景就是 52 步遍历。SKILL.md 的错误处理路径存在但不是最优路径。需要优化文档。
- **恢复失败 + 消耗低：** Agent 快速报错退出。从用户体验看不好，但至少没浪费 token。可能是 SKILL.md 没有写错误恢复路径。
- **恢复失败 + 消耗高：** 最差情况。Agent 反复尝试、疯狂重试、最终还是失败了。大量 token 浪费 + 用户等了很久 + 任务还是没完成。

**具体计算方式：**

给 D3 和 D5 分别打分，然后算加权综合分：

```
D3_score = 恢复成功次数 / 触发错误总次数 × 100
D5_score = min(1, 理论步骤数 / 实际步骤数) × 100  # 上限100，不倒扣

# 联动得分：D3 是前提，D5 是效率修正
combined = D3_score × (0.6 + 0.4 × D5_score / 100)
```

含义是：如果恢复失败（D3=0），无论 D5 多好综合分都是 0。如果恢复成功（D3=100），D5 的好坏在 60~100 分之间调节。这确保了"能恢复"比"恢复得快"更重要，但"恢复得快"仍然有区分度。

至于"该不该让 Agent 尽早失败而不是无限重试"——这不是评测框架该做的决策，而是 SKILL.md 该写的策略。评测框架的职责是**如实反映** Agent 的行为（恢复了但花了 50 步），让 SKILL.md 的维护者看到数据后自己决定：是加一句"最多重试 3 次后报错退出"，还是优化恢复路径让步骤降下来。

## 总结

面试官说得对：当前产出是一个"结果断言器"，不是"诊断中枢"。但结果断言器是诊断中枢的第一步——先有了能自动跑、能出报告的 pipeline，才有基础去加 Trace 解析、加 Mock 层、加多轮编排。

产出文档的问题在于把"已实现的能力"和"方法论设计"混在一起展示了，没有诚实地画出分界线。这个教训记住了。
